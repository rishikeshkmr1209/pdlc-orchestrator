# Contributing to the Claude Master Plugin

All engineering teams are welcome to contribute new skills, subagents, and slash commands. This document explains the full contribution process.

---

## Who Can Contribute

Any team member with a GitHub account in the the organization. External contributors are not accepted.

**Core maintainers** (required approvers): Platform Engineering team team. At least one core maintainer approval is required on every PR.

---

## What You Can Contribute

| Contribution Type | Template | Review Level |
|-------------------|----------|-------------|
| New skill | `templates/skill-template/SKILL.md` | Standard (2 approvals) |
| New subagent | `templates/agent-template.md` | Standard (2 approvals) |
| New slash command | Duplicate an existing `commands/*.md` | Standard (2 approvals) |
| Reference document | Add to a skill's `references/` directory | Standard (2 approvals) |
| Skill with Bash access | Same as above | **Security review required** |
| Skill with Write access | Same as above | **Security review required** |
| Bug fix / typo | Direct edit | 1 approval (any maintainer) |

---

## Naming Conventions

- **File names:** `kebab-case` only. No spaces, no underscores, no capitals.
  - Good: `aws-lambda-deploy.md`, `check-bundle-size.md`
  - Bad: `AWS_Lambda.md`, `checkBundleSize.md`
- **Skill names (frontmatter `name:`):** `kebab-case`, max 64 characters, no brand names.
  - Good: `generate-openapi-docs`, `check-ts-coverage`
  - Bad: `DataDog-monitoring`, `MySQLQueryOptimizer`
- **Agent names:** same rules as skill names, descriptive of the role.
- **Command aliases:** single lowercase word or hyphenated phrase matching the skill name.

## Step-by-Step Process

### 1. Check for existing coverage

Before building, search existing skills and agents to avoid duplication:

```bash
grep -r "name:" skills/ agents/
```

### 2. Start from a template

```bash
# For a new skill:
cp -r templates/skill-template/ skills/your-skill-name/

# For a new agent:
cp templates/agent-template.md agents/NN-your-agent-name.md
# Replace NN with the next available number (e.g., 07)
```

### 3. Fill in the template

- Complete **all frontmatter fields** — no empty required fields.
- Keep the SKILL.md body under **300 lines**. Move detail into `references/` or `templates/` subdirectories.
- Specify the **minimum required tools**. Don't request `Bash` or `Write` unless truly necessary.
- Add **3–5 evaluation queries** in a `## Evaluation` section at the bottom of SKILL.md (see below).

### 4. Write evaluation queries

Every skill and agent must include an `## Evaluation` section with test scenarios:

```markdown
## Evaluation

| Scenario | Input | Expected behavior |
|----------|-------|-------------------|
| Trigger — positive | "review my TypeScript file" | Skill activates, runs full checklist |
| Trigger — negative | "what time is it?" | Skill does NOT activate |
| Edge case | Empty file | Gracefully handles, returns "nothing to review" |
| Security boundary | Request to skip linting | Skill declines, explains why |
```

Minimum: 1 positive trigger, 1 negative (no-trigger), 1 edge case.

### 5. Open a pull request

- **Branch name:** `<JIRA-ID>-add-<skill-or-agent-name>` — e.g., `TICKET-1234-add-bundle-size-skill`
- **PR title:** `<JIRA-ID>: Add <skill-name> skill` — e.g., `TICKET-1234: Add bundle-size skill`
- A Jira ticket ID is **required** in both the branch name and PR title (security team requirement).
- Fill in the **full standard PR description template** (generated by the `create-pr` skill). All sections required.
- Self-check the security checklist in `docs/SECURITY-REVIEW.md` and paste results into the PR description if the skill uses `Bash` or `Write`.

### 6. Review requirements

| PR type | Approvals required | Security checklist |
|---------|-------------------|-------------------|
| New skill / agent (read-only tools) | 2 (≥1 core maintainer) | Not required |
| New skill with `Bash` or `Write` | 2 (≥1 core maintainer) | **Required** |
| Documentation update | 1 (any maintainer) | Not required |
| Bug fix | 1 (any maintainer) | Not required |

## Tool Least-Privilege Rules

When specifying `tools:` in your skill or agent frontmatter, follow this table:

| Task | Allowed tools |
|------|--------------|
| Reading and analyzing code | `Read, Grep, Glob` |
| Generating new files | `Read, Write, Glob, Grep` |
| Editing existing files | `Read, Edit, Glob, Grep` |
| Running shell commands | `Bash` — **requires security review** |
| Creating GitHub PRs | `Bash` — limited to `gh` CLI only |
| Full file system operations | Not permitted in shared skills |

Never list tools you don't use. Reviewers will flag unnecessary tool access.

---

## Security Checklist (self-review before submitting)

For any skill or agent that requests `Bash` or `Write` tool access, answer these questions before submitting:

- [ ] Does the skill validate all inputs before passing to shell commands?
- [ ] Are shell commands limited to a specific, documented allow-list?
- [ ] Does the skill avoid reading or writing outside the project directory?
- [ ] Are there no hardcoded credentials, URLs, or org-specific values?
- [ ] Is there no command that could exfiltrate data outside the internal network?
- [ ] Has the skill been tested against adversarial inputs (prompt injection attempts)?

See `docs/SECURITY-REVIEW.md` for the full checklist.

---

## Contribution Anti-Patterns (will be rejected)

- Skill bodies over 300 lines without a `references/` directory
- Hardcoded AWS account IDs, regions, or internal URLs
- `tools: ["*"]` or unconstrained tool lists
- Skills that duplicate functionality of an existing skill without meaningful improvement
- Missing `## Evaluation` section
- Agent `description:` that is vague (e.g., "helps with coding") — must be specific enough to trigger correctly
